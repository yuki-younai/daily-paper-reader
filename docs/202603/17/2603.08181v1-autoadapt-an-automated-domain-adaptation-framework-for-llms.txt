# AutoAdapt: An Automated Domain Adaptation Framework for Large Language Models

## Abstract 

Large language models (LLMs) excel in open domains but struggle in specialized settings with limited data and evolving knowledge. Existing do-main adaptation practices rely heavily on manual trial-and-error processes, incur significant hyper-parameter complexity, and are highly sensitive to data and user preferences, all under the high cost of LLM training. Moreover, the interactions and transferability of hyperparameter choices across models/domains remain poorly understood, mak-ing adaptation gains uncertain even with substan-tial effort. To solve these challenges, we present 

AutoAdapt , a novel end-to-end automated frame-work for efficient and reliable LLM domain adap-tation. AutoAdapt leverages curated knowledge bases from literature and open-source resources to reduce expert intervention. To narrow the search space, we design a novel multi-agent debating system in which proposal and critic agents iter-atively interact to align user intent and incorpo-rate data signals and best practices into the plan-ning process. To optimize hyperparameters under tight budgets, we propose AutoRefine , a novel LLM-based surrogate that replaces costly black-box search. Across 10 tasks, AutoAdapt achieves a 25% average relative accuracy improvement over state-of-the-art Automated Machine Learn-ing baselines with minimal overhead.

LLMs often fall short when applied to specialized domains where data is scarce, proprietary, or subject to privacy con-straints. Further, in scenarios where data evolves with time, pretrained LLMs would not have the updated knowledge. In such settings, domain adaptation of LLMs is imperative by resorting to techniques like in-context prompting (Dong et al., 2024) using retrieval from current data (RAG) (Lewis et al., 2020) or finetuning the models (SFT) (Wang et al., 2025) on new data for optimal performance. The current process of adapting LLMs to a specific task is largely man-ual and remains limited to a set of best practices, assuming a one-template-fits-all approach across tasks and models. Further, effective fine tuning relies on expert knowledge be-cause small choices in data, objectives, and training settings can strongly affect the performance. Domain adaptation for LLMs involves a series of intricate steps: curating domain-specific datasets, selecting appro-priate model architectures, configuring prompting or fine-tuning strategies, and iteratively optimizing hyperparame-ters for optimal performance. Each decision point, rang-ing from model selection, context engineering, training technique (e.g., supervised fine-tuning (SFT), parameter-efficient adaptation (PeFT) e.g. LoRA/Prefix Tuning, pref-erence alignment e.g. DPO) to the choice of learning rates, batch sizes, and regularization techniques, can significantly impact the final model’s quality, stability and cost. The challenge of exploring the vast search space is compounded in LLMs due to the high time and cost of finetuning these large models. The combinatorial explosion of possible con-figurations, coupled with the high time and resource utility, means that even experienced teams must engage in extensive trial-and-error, often spending weeks or months to achieve robust, high-quality results. Recent research (Guo et al., 2024; Trirat et al., 2025; Zhang et al., 2024; 2023) has explored automating general Machine Learning (ML) pipelines using agent-based frameworks of-fering partial solutions for model selection and hyperparam-eter tuning, albeit in the general ML domain. However, the following challenges exist : 1) Existing methods either rely on the pretraining knowledge or in-context learning ability of LLMs, which we show is insufficient for the task of LLM adaptation. To address these challenges, we introduce AutoAdapt , an automated domain adaptation framework that proposes to solve the adaptation problem as a multi-agent system grounded in a knowledge-driven hierarchical workflow. We further use Gaussian processes (Beckers, 2021) for proba-bilistic function estimation, modeling uncertainty over the adaptation objective function. AutoAdapt leverages a structured directed acyclic graph, called “Adaptation Configuration Graph” (ACG), that in-corporates the LLM adaptation workflow knowledge. This graph contains hierarchical decision nodes (e.g. RAG vs 

SFT , SFT using LoRA or Prefix Tuning etc.) and parameter nodes (e.g. learning rate, epochs, batch size to explore for SFT ). The framework is com-posed of a system of proposal and critic agents to generate, debate, verify and refine executable pipelines. We ground the multi-agent system by integrating best practices from thousands of open-source models and datasets, as well as user-specified requirements and data characteristics. We further perform online retrieval from web sources and liter-ature. Thus, AutoAdapt dramatically reduces the need for manual intervention and expert oversight. This produces an adaptation pipeline with narrowed search spaces (e.g. 

1e−4 ≤ learning rate ≤ 1e−6) to optimize. The framework further incorporates a feedback-driven optimiza-tion loop, called AutoRefine , to optimize parameters (e.g. ex-act SFT parameters like learning rate = 2 e−5 from the narrowed search space). AutoRefine efficiently explores the hyperparameter space using LLM-based oracle that fac-tors performance from prior runs. However naive usage of LLMs to predict performance (surrogate modeling) (Liu et al., 2025) in the multi-dimensional LLM hyperparameter space, is prone to hallucination. We propose to solve this by grounding the LLM surrogate with function estimation techniques. Through extensive empirical evaluation across diverse do-mains, including mathematical and general reasoning, medi-cal question answering, legal analysis, programming, agent tool calling, e-commerce, and real world cloud incident diagnosis, we demonstrate that AutoAdapt consistently out-performs existing AutoML baselines, delivering significant accuracy gains with minimal additional computational over-head (w.r.t. default choices). By automating the end-to-end LLM adaptation process , AutoAdapt makes robust, domain-specific LLMs accessible to a broader range of users and organizations, bridging the gap between cutting-edge re-search and practical real-world applications . ## 2. Related Works 

There exists a plethora of works in literature on automated machine learning. Here, we focus on agentic solutions to automate this process. In agentic literature, prior works such as (Zhang et al., 2023; Shen et al., 2023; Hong et al., 2024; Liu et al., 2025) focus on specific aspects of the ML training pipeline such as data preparation, analysis or HPO etc. MLCopilot (Zhang et al., 2024) was one of the first agentic systems tackling general ML pipelines, using an offline stage to build knowledge from experience and an online retrieval stage. Similarly DS-Agent (Guo et al., 2024) proposes to use LLM agents along with case based reasoning (CBR) to come up with generic data science pipelines for a task. AutoML-Agent (Trirat et al., 2025) employs a multi-agent LLM framework using retrieved knowledge bases to automate generic ML training pipelines. These methods are not tailored to LLMs and fall short in reliably planning and executing complex, multi-stage LLM adaptation workflows. Unlike prior methods that typically support only single-stage training pipelines, AutoAdapt provides a comprehensive framework for full domain adaptation pipelines, including multi-stage processes such as in-context prompting (e.g. RAG), supervised fine-tuning (SFT), alignment (DPO) and so on. AutoAdapt aims to reliably and efficiently generate user-aligned executable LLM adaptation plans, overcoming the limitations of existing AutoML systems. Please refer the Appendix B for further details.

Proposition 3.1. Let H(c) be the feasible configuration space and P(c) = P1 × · · · × P T the prior -guided sub-space. Denote the expected task loss as LE(h), the op-timal loss over the full space is L∗H = min h∈H (c) LE(h)

and the optimal loss over the prior - guided space is L∗P =min h∈P (c) LE(h). Considering LE(h) is Lipschitz continu-ous in hyperparameters, the prior -guided subspace satisfies the generalized ε-covering bound: 

L∗P ≤ L∗H + Lε, 

where L is the Lipschitz constant and ε is the distance between the optimal parameters in H(c) and P(c).

The above result is straightforward to verify after continu-ous relaxation (Boyd & Vandenberghe, 2004) of the discrete space. It implies that restricting search to the prior -guided subspace incurs at most a controlled additive suboptimal-ity while yielding dramatically improved tractability. For the optimization over the constrained search space we pro-pose to use LLMs together with function estimation (using Gaussian process) techniques. More details on Gaussian processes can be found in Appendix A and Beckers 2021. Note, the above formulation is intended solely for clarity of exposition and the proposed method does not depend on the stated assumptions.

The multi-agentic system needs to be grounded in the real world LLM usage ecosystem. 4.2. Adaptation Configuration Graph (ACG) 

The ACG formalizes adaptation planning as a directed acyclic graph. The nodes are of two types: selection nodes for discrete strategy/technique (e.g. RAG vs SFT ) choices and parameter nodes for hyperparameters within the selected technique (e.g. learning rate in SFT ). 

Construction: We construct the ACG manually based on best practices. The start selection node selects between 

RAG , SFT and Alignment . Along SFT , the decision path includes PEFT and within PEFT the choices include 

{LoRA, DoRA, Prefix Tuning} . The selection nodes for Alignment can be invoked post SFT or without 

SFT (using a pretrained model). The choices inside 

Alignment include {DPO, RLHF} . The Parameter nodes, present at the leaf of a traversed path, represent the decision for the continuous/categorical parameter ranges for e.g. parameters for SFT include epochs, learning rate, LoRA rank/alpha/dropout, batch size, gradient accumulation, quantization etc. Edges encode dependencies e.g. choosing LoRA in PEFT selection node activates the dependent Parameter node to decide the search space for 

{lora_r, lora_alpha, lora_dropout} .

Hierarchical Decision over the ACG: Figure 3 shows a sample ACG workflow. The planner (multi-agent debate system outlined next) starts at high - level decision nodes (e.g., RAG , SFT and Alignment ) and traverses until leaf nodes (complete executable plan). At each node, the planner retrieves information from the best - practice KB and aggre-gates them with user constraints (compute/time/budget) to decide the pipeline. This structured approach constrains search making the adaptation efficient and ensures reliable execution. The traversed path need not adhere to a linear chain graph but could also involve multiple paths e.g. SFT 

followed by Alignment or RAG as decided by the system. 

4.3. Multi-Agent Debate Planner 

In order to make the decision at the nodes of the ACG, we design a multi-agent pipeline. The selection nodes of the ACG could also be treated as hyperparameters and evaluated. However this increases the search space which becomes infeasible for the case of LLMs. Thus we need an efficient planning system akin to expert humans selecting techniques from knowledge or literature. The multi - agent system consists of proposal and critic agents that debate in turns to come up with the final decision on the technique, model and parameter ranges. Below we briefly describe the agents (with details in Appendix E) 

Proposal Agents: Proposal agents are responsible for con-structing and iteratively refining candidate domain adapta-tion pipelines. These agents are (i) best practice agent : con-solidates empirical knowledge from prior training pipelines, model cards, and published benchmarks, and (ii) knowledge retrieval agent : dynamically retrieves evidence from recent literature, technical reports, and online resources. (iii) re-search agent : utilizes the research information retrieved from arxiv, KBs etc. to propose techniques. These agents initialize the search process in regions of the configuration space that historically yield strong performance while re-maining responsive to emerging techniques. 

Critic Agents: Critic agents verify the proposed configu-rations against user, data and resource constraints. (i) The 

user agent enforces user - specified requirements, such as compute budgets, training preferences, methodological re-strictions, and domain - specific constraints. In parallel, (ii) the data agent performs data -centric critique by analyzing statistics of the target dataset, including scale, token dis-tributions, domain characteristics etc., and mapping these signals to concrete constraints on adaptation choices. After each round of debate, an aggregator agent aggregates the proposals and critiques. The proposal agents further revise the proposals based on the critiques and unsatisfied constraints. In this manner, each ACG node undergoes 

proposal −→ critique −→ revision iterations. The loop termi-nates when all active constraints are satisfied and the plan is deemed executable (with runnable code). 4.4. AutoRefine 

This module decides the optimal hyperparameter set using an LLM surrogate method. Prior works (Zhou et al., 2025) have shown that transformers (and thus LLMs) have lim-itations in time series forecasting. This issue is bound to be amplified in the multivariate forecasting by works us-ing LLMs as surrogates (Liu et al., 2024) to predict results. Thus we propose a principled approach grounded in func-tion estimation theory to guide the surrogate modeling from prior knowledge of historic runs. We use Gaussian Process (GP) model for function estimation. Algorithm 1 provides a pseudo-code for AutoRefine. We start by sampling a coreset (Jubran et al., 2019) of the data for efficient execution. We then perform few ( ∼ 3) execu-tion runs on the selected pipeline (e.g. SFT ) by invoking an LLM to sample initial parameters from the narrowed search space (lines 2-5). Inspired by (Requeima et al., 2024), we prompt the LLM to predict the results along each parameter dimension from the evaluated points (lines 9-12). We reason that predicting the trend along each parameter axis will help reduce hallucinations compared to directly estimating the results on the higher dimensional space. These historic and predicted results are used to train a GP model to obtain a function estimation of the remaining points in the search space (line 13). Note the GP model requires hundreds of data points to learn the function and it is not possible to finetune/evaluate LLMs for many iterations. Hence, we ob-tain the parameter-axis wise trends of the results to augment data. Once the model of the results over the search space is obtained we need an acquisition function that would max-imize the result of the acquired hyperparameter set. For acquisition, we find the optimal point of function using LBFGS (Liu & Nocedal, 1989) (after continuous relaxation of the categorical variables), starting from each of the his-toric evaluated samples (lines 14-18). Post discretization, the LLM is prompted in-context with these optimal points to choose the next parameter set to evaluate on (lines 19-21), balancing exploration and exploitation. Thus the LLM surrogate is guided by the optimal hyperparameter as per the estimated function that is grounded in results from prior runs, reducing hallucinations. We allow a maximum of B

(loops starting line 2,7) function evaluations (e.g. finetun-ing followed by testing on validation set) on the selected parameters. ## Impact Statement 

AutoAdapt aims to lower the technical and resource barriers to domain adaptation of large language models by automat-ing pipeline design, configuration selection, and refinement. By enabling efficient customization using smaller datasets and constrained budgets, AutoAdapt can reduce redundant experimentation, lower computational cost, and democra-tize access to high - quality domain -specific models for re-searchers and practitioners with limited resources. This can benefit applications in areas such as healthcare, education, legal analysis, and scientific research, where domain exper-tise exists but large -scale machine learning infrastructure may not. At the same time, AutoAdapt inherits the risks associated with adapting LLMs to sensitive domains, including the potential amplification of dataset biases or inappropriate behaviors if user - provided data is flawed or unrepresentative. While AutoAdapt does not generate new data on its own in current state, its automation could accelerate deployment of domain -adapted models without sufficient human review. To mitigate these risks, AutoAdapt is explicitly designed to surface configuration decisions, maintain modularity across stages, and enable human oversight through interpretable planning stages. We view AutoAdapt as a decision - support system rather than a replacement for responsible model building evaluation. Overall, AutoAdapt contributes to more efficient and trans-parent LLM adaptation workflows, with the potential to re-duce environmental impact through compute savings while emphasizing the importance of careful downstream evalua-tion and governance in real-world deployments.

## A. Preliminaries 

Multi-agent frameworks have emerged as a powerful paradigm for complex decision-making tasks, enabling distributed reasoning through specialized agents that collaborate via a plethora of communication workflows. In the context of automated machine learning and domain adaptation, these frameworks allow decomposition of pipeline design into modular decisions, improving scalability and adaptability compared to monolithic approaches. 

A.1. Bayesian Optimisation 

Bayesian optimization (BO) (Frazier, 2018) is a sequential model-based approach for optimizing expensive black-box functions. It assumes an unknown objective function f (x) and models it using a probabilistic surrogate, typically a Gaussian Process (GP) (Beckers, 2021). The GP defines a prior over functions and updates it with observed data Dt = {(xi, y i)}ti=1 .The posterior predictive distribution at a new point x∗ is: 

μ(x∗) = k(x∗, X )⊤K−1y,σ2(x∗) = k(x∗, x∗) − k(x∗, X )⊤K−1k(X, x∗)

where K is the kernel matrix with entries k(xi, xj ).The next query point is chosen by maximizing an acquisition function α(x), which balances exploration and exploitation. Common choices include: 1) Expected Improvement (EI): αEI (x) = E[max(0 , f (x) − f +)] = ( μ(x) − f +)Φ( z) +

## B. Related Works 

Recent advances in LLM-powered AutoML frameworks can be broadly categorized into static and dynamic systems. Static recommendation approaches, such as AutoMMLab (Yang et al., 2024) , select from a set of pre-prepared models and produce executable plans with limited adaptability to user-specific requirements. Dynamic systems, such as MLCopilot (Zhang et al., 2024), DS-Agent (Guo et al., 2024) and AutoML-Agent (Trirat et al., 2025), generate training plans and code , offering better alignment with user tasks but often struggling with reliability and the complexity for long, multi-stage adaptation pipelines. 

Existing AutoML agents : AutoGPT (Zhang et al., 2023) and HuggingGPT (Shen et al., 2023) connect LLMs with external tools and models to solve generic AI tasks. Agents such as (Guo et al., 2024; Hong et al., 2024; Liu et al., 2025) focus on specific aspects of the ML training pipeline such as data preparation, analysis or HPO and do not consider the end to end pipeline for LLM adaptation. MLCopilot (Zhang et al., 2024) uses a two-stage process: an offline stage to build a knowledge pool from historical data and experience, and an online stage to retrieve demonstrations and generate solutions based on new task descriptions. DS-Agent proposes to use LLM agents along with case based reasoning (CBR) to come up with generic data science pipelines for a task. AutoML-Agent (Trirat et al., 2025) is the closest work related to our method. It employs a multi-agent LLM framework using retrieved knowledge bases to automate generic ML training pipelines. However, the method is not tailored to adaptation of LLMs which may not require finetuning (RAG, prompt optimization 12 AutoAdapt: An Automated Domain Adaptation Framework for LLMs 

etc.) or go beyond finetuning (multi-stage RLHF, instruction tuning etc.). These methods fall short in reliably planning and executing complex, multi-stage LLM adaptation workflows. Different from the above works, AutoAdapt introduces a hybrid recommendation approach that combines the reliability of static planning (using ACG) with the adaptability of dynamic systems (multi-agent system guided by best practices KB). Unlike prior methods that typically support only single-stage fine-tuning, AutoAdapt provides a comprehensive framework for full domain adaptation pipelines, including multi-stage processes such as in-context prompting (eg. RAG), supervised fine-tuning (SFT), alignment (DPO) and so on. AutoAdapt aims to reliably and efficiently generate executable, robust, and user-aligned LLM adaptation plans, overcoming the limitations of existing AutoML systems.

## D. Adaptation Configuration Graph 

In this section, we describe in detail the design of the ACG utilized to run the multi agent planning pipeline. Figure 8 illustrates the Adaptation Configuration Graph (ACG) used in our framework to structure and constrain the space of large language model (LLM) domain adaptation workflows. The ACG is a directed acyclic graph that explicitly encodes high -level adaptation strategies, their dependencies, and the corresponding parameterized training configurations, enabling systematic and reliable planning of multi-stage adaptation pipelines. At the highest level, the ACG branches from a root adapt node into alternative adaptation strategies, including retrieval -augmented generation (RAG), supervised fine -tuning (SFT), and preference - based alignment (DPO). Each strategy node captures a semantically distinct mode of adaptation and is connected via requires edges to the concrete configuration blocks needed to instantiate that strategy. This design allows the planner to reason explicitly over which adaptation tech-niques are applicable under given data, resource, and user constraints, rather than treating the pipeline as an unstructured sequence of hyperparameters. For RAG, the graph specifies the required retrieval configuration, including the embedding model, retriever parameters (e.g., top -k), and the base model used for generation. This separation allows retrieval - specific choices to be optimized independently of downstream training decisions. For SFT and DPO, the ACG links the corresponding strategy nodes to detailed training configurations that define the base model, optimization hyperparameters (learning rate, batch size, gradient accumulation), logging and evaluation intervals, and training duration. By explicitly modeling these configurations as graph nodes, the ACG exposes the hierarchical structure of LLM adaptation decisions. 16 AutoAdapt: An Automated Domain Adaptation Framework for LLMs 

Figure 8. Adaptation Configuration Graph (ACG) for the LLM adaptation pipeline with example parameter values at each parameter node. 

The graph further captures parameter - efficient fine - tuning (PEFT) as an intermediate abstraction. Both SFT and DPO may route through a PEFT node, which in turn branches into concrete techniques such as LoRA and prefix tuning. Each PEFT variant is associated with its own parameter block, specifying method - specific hyperparameters (e.g., LoRA rank and scaling factors, prefix length) as well as shared efficiency settings such as quantization type. This explicit factoring enables the planner to reuse shared structure while still exploring method-specific configurations. Overall, the ACG serves two key purposes. First, it narrows the effective search space by enforcing valid structural dependencies between adaptation stages and parameters, thereby preventing ill - formed or non - executable pipelines. Second, it provides a transparent and interpretable representation of the adaptation workflow, which can be traversed and refined by downstream optimization components. By grounding pipeline construction in the ACG, our framework achieves reliable execution while retaining flexibility to support diverse adaptation strategies and training regimes.

## G. Best-Practices Knowledge Base 

In this section we explain the construction of the best practices KB and provide studies on the best practices retrieved and the consistency and relevance of the recommendations provided using these. 

G.1. KB Construction 

To ground the adaptation planning process in empirical evidence and widely adopted practices, we construct a Best -Practices Knowledge Base (KB) that aggregates and structures information from diverse public sources on large language model (LLM) training, adaptation, and evaluation. The primary goal of the KB is to expose reliable priors over model choices, adaptation strategies, and hyperparameter configurations, which can be leveraged by the multi - agent planner to generate executable and effective adaptation pipelines. 

Knowledge Sources We curate the KB from multiple complementary sources that reflect both community standards and state -of - the - art empirical results. Specifically, we collect information offline from: (i) model cards and dataset cards hosted on Hugging Face, which provide structured descriptions of training regimes, datasets, parameter choices, evaluation metrics, and reported performance; (ii) fine -tuning code repositories and notebooks from platforms such as GitHub and Unsloth, which expose concrete implementation details and hyperparameter settings used in practice; and (iii) leaderboards and evaluation summaries from Papers with Code, which link datasets to top - performing models and report comparative metrics. These sources provide complementary perspectives, ranging from high-level summaries to low-level configuration details. 

Knowledge Extraction and Structuring Raw information from the above sources is heterogeneous and largely unstructured. To address this, we employ an LLM -based extraction pipeline to convert documents and code into structured representa-tions. For each model or dataset entry, we extract salient attributes such as dataset domain, size, task type, input–output characteristics, model family, parameter count, training pipeline (e.g., SFT, PEFT, DPO), and reported hyperparameter ranges. In the case of code repositories, we further extract explicit parameter instantiations (e.g., learning rate, batch size, number of epochs, PEFT - specific settings) from training scripts and notebooks. All extracted items are normalized and stored as structured records. We manually validate a handful of the extracted data, however due to the scale of the data manual verification of the entire KB is left to future works.

G.2. Dispersion Index of Recommended Parameter Ranges 

To quantify the diversity of hyperparameter range recommendations produced by our multi-agent framework, we introduce the Dispersion Index (DI) . The DI captures the degree of variability across a set of configurations by jointly considering categorical and numeric dimensions. Unlike pairwise similarity measures that focus on individual comparisons, our metric summarizes the overall spread of the recommendation set, providing a single interpretable score. Formally, let C = {c1, c 2, . . . , c N } denote the set of recommended configurations. Each configuration comprises categorical parameters (e.g., activation functions, optimizer choices) and numeric parameters (e.g., learning rate, batch size). For any pair (ci, c j ), we compute: Sim (ci, c j ) = α · J(ci, c j ) + (1 − α) · IoU (ci, c j ), (7) where J(·, ·) is the Jaccard similarity over categorical tokens, IoU (·, ·) is the weighted average Intersection-over-Union across numeric ranges, and α ∈ [0 , 1] balances the contributions of categorical and numeric components.
