## 摘要
用于代码生成的强化学习依赖于来自单元测试通过率的可验证奖励。然而，高质量的测试套件十分稀缺，现有数据集的覆盖范围有限，且静态奖励无法随着模型的改进而自适应。最近的自博弈方法将代码生成和测试生成统一在单个模型中，但面临一个固有的困境：白盒访问会导致自我勾结，即模型为了轻松获得奖励而生成琐碎的测试；而黑盒限制则会产生通用的测试，从而遗漏特定于实现的漏洞。我们提出了 Code-A1，这是一个对抗性协同演化框架，通过相反的目标共同优化代码大语言模型（Code LLM）和测试大语言模型（Test LLM）。代码模型因通过更多测试而获得奖励，而测试模型则因暴露更多缺陷而获得奖励。这种架构上的分离消除了自我勾结的风险，并安全地实现了白盒测试生成，使测试模型能够检查候选代码以构建有针对性的对抗性测试。我们还引入了一种用于经验回放的“错题本”机制，以及一种平衡测试有效性与对抗难度的复合奖励。在 Qwen2.5-Coder 模型上的实验表明，Code-A1 的代码生成性能达到或超过了在人工标注测试上训练的模型，同时显著提升了测试生成能力。

## Abstract
Reinforcement learning for code generation relies on verifiable rewards from unit test pass rates. Yet high-quality test suites are scarce, existing datasets offer limited coverage, and static rewards fail to adapt as models improve. Recent self-play methods unify code and test generation in a single model, but face a inherent dilemma: white-box access leads to self-collusion where the model produces trivial tests for easy rewards, yet black-box restriction yields generic tests that miss implementation-specific bugs. We introduce Code-A1, an adversarial co-evolution framework that jointly optimizes a Code LLM and a Test LLM with opposing objectives. The Code LLM is rewarded for passing more tests, while the Test LLM is rewarded for exposing more defects. This architectural separation eliminates self-collusion risks and safely enables white-box test generation, where the Test LLM can inspect candidate code to craft targeted adversarial tests. We further introduce a Mistake Book mechanism for experience replay and a composite reward balancing test validity with adversarial difficulty. Experiments on Qwen2.5-Coder models demonstrate that Code-A1 achieves code generation performance matching or exceeding models trained on human-annotated tests, while significantly improving test generation capability.

### Code-A1: 通过强化学习实现代码 LLM 与测试 LLM 的对抗演化 —— 论文总结

#### 1. 核心问题与整体含义（研究动机和背景）
代码生成的强化学习（RL）高度依赖可验证的奖励信号（单元测试通过率）。然而，该领域面临三大挑战：
*   **高质量测试稀缺**：人工标注的测试集规模有限且覆盖面不足。
*   **静态奖励滞后**：固定的测试用例无法随着模型能力的提升而动态调整难度。
*   **自博弈（Self-play）困境**：当单一模型同时生成代码和测试时，若允许“白盒”访问代码，模型会产生“自勾结”（生成极简测试以获取高分）；若限制为“黑盒”访问，生成的测试又过于通用，无法捕捉特定实现的逻辑漏洞。

**Code-A1** 的核心意义在于通过**对抗性协同演化**，打破上述僵局，实现代码生成与测试生成的双重进化。

#### 2. 方法论：核心思想与关键技术
Code-A1 采用双策略优化框架，将任务解耦为两个具有相反目标的模型：
*   **核心思想**：**代码模型（Code LLM）**的目标是编写能通过更多测试的代码；**测试模型（Test LLM）**的目标是编写能发现代码缺陷的对抗性测试。
*   **关键技术细节**：
    *   **架构分离**：通过物理分离两个模型，安全地开启“白盒测试”，即 Test LLM 可以观察候选代码的逻辑，从而精准打击其薄弱环节，同时避免了自勾结风险。
    *   **错题本机制（Mistake Book）**：建立每道题的经验回放缓冲区，记录历史失败的测试用例。这能防止代码模型在学习新知识时产生“灾难性遗忘”，并为奖励计算提供稳定的基准。
    *   **复合奖励设计**：
        *   **Code LLM 奖励**：基于历史测试（错题本）和新生成测试的平均通过率。
        *   **Test LLM 奖励**：平衡**有效性**（测试必须语法正确且预测准确）与**对抗难度**（如果新测试能让原本通过历史测试的代码失败，则奖励更高）。
    *   **算法流程**：使用 **GRPO（组相对策略优化）** 算法。在每个训练步中，Code LLM 生成多个解，Test LLM 针对每个解生成多组测试，通过沙箱执行后更新错题本并同步优化两个模型。

*   **Benchmark（基准测试）**：

#### 5. 实验数量与充分性

#### 6. 主要结论与发现
*   **性能超越人工**：Code-A1 训练的代码模型在多个榜单上达到或超过了使用人工标注数据训练的效果。
*   **测试能力跨级提升**：Code-A1 训练的 **3B 模型**在测试生成能力（Mul 分数）上超过了 **7B 的基础模型**，证明对抗演化比单纯堆参数更有效。
*   **协同效应**：在推理阶段（Test-time Scaling），Code-A1 训练的测试模型能更有效地从多个候选解中筛选出正确答案。
*   **动态平衡**：错题本和复合奖励成功稳定了对抗训练的波动，使模型进入了“更强的代码需要更难的测试，更难的测试驱动更强的代码”的正向循环。

#### 7. 优点
*   **解决自勾结**：通过架构分离巧妙解决了自博弈中的作弊问题。
*   **白盒优势**：允许测试模型“读代码”，使其生成的测试具有极强的针对性和 Debug 价值。
*   **经验回放**：错题本机制有效解决了 RL 中的遗忘问题，保证了训练的单调改进。
*   **数据效率**：无需额外的人工标注，仅靠对抗即可实现自我进化。

#### 8. 不足与局限
*   **依赖 Ground Truth 代码**：在训练阶段，验证测试用例的正确性仍需要参考答案（Oracle），这限制了其在完全无标注领域（如前沿科学发现）的应用。
*   **编程语言局限**：实验仅针对 Python 语言，且主要集中在函数级任务，尚未验证在大型项目级代码或多语言环境下的表现。
*   **测试格式限制**：目前仅支持 Assertion（断言）格式，无法处理涉及 I/O、状态机或复杂交互的测试场景。
*   **算力开销**：双模型同时训练及频繁的沙箱交互对计算资源和系统稳定性有较高要求。
